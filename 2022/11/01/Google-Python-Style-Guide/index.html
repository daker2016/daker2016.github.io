Python Language Rules

Imports

Tips

  • 対象：package、module
  • 対象外：class、function
  • ‘type’ Moduleは例外 →　例：from typing import Any, TypeVar

定義

  • モジュール間のコードの再利用メカニズム。 例：Module BにModule Aの内容を使いたい場合、Moduleをimportすれば、使えます

メリット

  • namespaceの管理規則は簡単です。
  • 各識別子元の指定方式は一致すること　例：x.Obj→　対象objはmodule xに定義されることを示している

デメリット

  • moduleの名前は競合することがある
  • あるmoduleの名前は長すぎで、ソースは読みずらいまたはフォーマットしにくい

結論

  • import x：packageとmoduleのimporting
  • from x import y：xはpackage名、yはmodeule名
  • from x import y as z：同時にimportされている二つ以上のmoduleの名前が重複またはmodule yの名前が長すぎの場合、別名zを付ける
  • import y as z：zは標準略称の場合のみを使う　例：import nump as np

例

  • sound.effects.echoをimportする

例外

  • 下記のModuleに定義されているSymbols（pythonのドキュメントではクラス名，関数名や変数名を総称して、名前とかシンボルと呼んでいるよう）はstatic analysis and type checkingのため、import module名ではなくて、from module名 import Symbol名にする必要があります
    • typing module
    • collections.abc module
    • typing_extensions module
  • six.movesmodule中にのリダイレクト　
    • six.movesはPython2とPython3のコード互換性を実現する
    • Python3は標準ライブラリを再編成、一部functionsを別moduleに移動しました。six.moves moduleは一貫性のあるインタフェースを提供して、Python2とPython3のコード互換性を保障する.Importの：from six.moves import urllib
    • stackoverflowを参照してください。

Exceptions

Tips

  • Exceptionsが使えますが、注意が必要です

定義

  • Exceptionsは正常のコントロールフローを停止して、エラーまたは他の異常条件を処理すること

メリット

  • 通常の操作コードの制御フローは、エラー処理コードによって乱雑になりません。 また、
  • 特定の条件が発生したときに、制御フローが複数のフレームをスキップすることもできます。たとえば、エラー コードを配管する代わりに、N 個のネストされた関数から 1 つのステップで戻ることができます。

デメリット

  • 処理フローを難しくになって理解しにくい可能性がある
  • ライブラリを呼び出すときに、エラーケースは見落としやすい

結論

  • 下記の条件に従う必要がある
    • 優先的に、適当にbuilt-inのexceptionクラスを使う。例えば、値が想定外になる場合、ValueErrorをraiseする。パラメータのバリデーター処理にassertを使わないでください。assertは内部ロジックが正しいことを検証するために使う。パラメータチェック異常とかは使わない。どうしてもexceptionで異常を示したい場合、assertではなく、raise statementを使う
    • ライブラリ、パッケージとかは内部exceptionが定義できます。定義するときに、既に存在しているExceptionクラスを継承しないといけない。Exeptionの名前の最後にError付ける必要がある
    • catch-allexcept:、Exception、StandardErrorは絶対に使わないでください。except:なら、すべてのエラー、typo、sys.exit()、Ctrl+Cなどはcatchしている。下記のケースは例外として、使っても大丈夫です
      • このexceptionを外の処理に渡す　例： A処理中身にB処理があって、B中にcatchして、Aにこのexceptionをraiseする
      • 例外が伝播されずに記録されて抑制されるプログラム内に分離ポイントを作成します。スレッドAにExceptionが発生して、他のスレッドとかシステム的に影響がないこと
    • try/except内部おソースできる限りに少なめに書いたほうがいい。ソースが多くと、想定外のexceptionが発生する可能性がある。例えば、catch ValueErrorですが、中にConnectionErrorが発生したら、catch処理にこのケースは対応できない
    • 最後にfinally処理を追加してください。リソースの解放とかはこの中にやったほういい。

Global variables

Tips

  • Global変数を使わないでください。

定義

  • moduleレベルの変数またはクラスのattribute変数

メリット

  • 偶に役に立つ

デメリット

  • moduleの動作を変わる可能性があります。importするときに、Global変数に値を設定処理があるため　※マルチスレッドでthreadsafeの問題もあるかもしれません

結論

  • Global変数の使用を避けたほうがいい
  • どうしても必要な時に、moduleレベルの変数として定義して、名前に_のprefixを付けて、外部にこの変数をアクセスしたい場合、public module関数を用意して呼び出すようにしたほうがいい
  • module-level定数は推奨です。例：_MAX_HOLY_HANDGRENADE_COUNT = 3。定数名前は大文字とアンダーバーで定義する必要がある。

Nested/Local/Inner Classes and Functions

Tips

  • ネストされたローカル関数またはクラスは、ローカル変数を閉じるために使用する場合に問題ありません。 内部クラスは問題ありません。

定義

  • classはmethod、function、class内部に定義できる
  • functionはmethod、function内部に定義できる
  • ネストされた関数はenclosing scops内定義された変数に対して、read権限しかない

メリット

  • 一定範囲内しか使えないユーティリティクラスと関数が定義できる。decorator中に良く使われている

デメリット

  • ネストされた関数とクラスは直接にテストできない。ネスト処理は関数を長くになって、読みにくいかもしれません

結論

  • 使用するときに注意点がある。self または cls 以外のローカル値を閉じる場合を除いて、ネストされた関数またはクラスは避けてください。モジュールのユーザーから隠すためだけに関数をネストしないでください。 代わりに、モジュール レベルでその名前の前に _ を付けて、テストから引き続きアクセスできるようにします。

Comprehensions & Generator Expressions

Tips

  • 簡単のケースは使います

定義

メリット

  • comprehensionsは他の方法より簡単かつ分かりやすい
  • Generatorはlist作成を避けるため、効率は良くになる

デメリット

  • 複雑なcomprehensionsとGeneratorは読みにくい

結論

  • 簡単のケースは使う。
  • mapping expression、for clause、filter expressionは一行に書く
  • 複数のfor処理またはfilterのケースは使わないほうがいい。forループにしたほうがいい

Generators

Tips

  • 必要な場合、Generatorsを使います

定義

  • ジェネレーター関数は、yield ステートメントを実行するたびに値を生成する反復子を返します。 値を生成した後、次の値が必要になるまで、ジェネレーター関数の実行時の状態は中断されます。
  • Generators:https://qiita.com/keitakurita/items/5a31b902db6adfa45a70

メリット

  • ローカル変数の状態と制御フローが呼び出しごとに保持されるため、コードが単純になります。 ジェネレーターは、一度に値のリスト全体を作成する関数よりも少ないメモリを使用します。

デメリット

  • 基本はない

結論

  • 推奨です。docstring 作成する時にReturns:よりYields:を使ったほうがいい
  • ジェネレーターが高価なリソースを管理している場合は、必ずクリーンアップを強制してください。
  • クリーンアップを行う良い方法は、ジェネレーターをコンテキストマネージャーPEP-0533でラップすることです。

Lambda Functions

Tips

  • 1行ならOKです。ラムダを使用した map() または filter() よりもジェネレーター式を優先する

定義

  • ラムダは、ステートメントではなく式で匿名関数を定義します。

メリット

  • 便利

デメリット

  • ローカル関数よりも読み取りとデバッグが困難です。 名前がないということは、スタック トレースが理解しにくいことを意味します。 関数には式しか含まれないため、表現力が制限されます。

結論

  • ワンライナーに使用しても大丈夫です。 ラムダ関数内のコードが 60 ～ 80 文字を超える場合は、通常のネストされた関数として定義する方がよいでしょう。
  • 乗算などの一般的な演算には、ラムダ関数の代わりに operator モジュールの関数を使用します。 たとえば、ラムダ x, y: x * y よりも operator.mul を優先します。

Conditional Expressions

Tips

  • シンプルなケースでOK。

定義

  • 条件式 (「三項演算子」と呼ばれることもあります) は、if ステートメントの構文を短くするメカニズムです。 例: x = 1 if cond else 2.

メリット

  • if 文よりも短くて便利です。

デメリット

  • if 文より読みにくいかもしれません。 式が長い場合、状態を特定するのが難しい場合があります。

結論

  • 単純な場合に使用しても問題ありません。 true-expression、if-expression、else-expression の各部分は 1 行に収まる必要があります。 より複雑な場合は、完全な if ステートメントを使用してください。

Default Argument Values

Tips

  • 多数の場合は大丈夫です。

定義

  • 関数のパラメータリストの最後に変数の値を指定できます (例: def foo(a, b=0):)。 foo が 1 つの引数だけで呼び出された場合、b は 0 に設定されます。2 つの引数で呼び出された場合、b は 2 番目の引数の値を持ちます。

メリット

  • 多くの場合、多くのデフォルト値を使用する関数がありますが、まれにデフォルト値をオーバーライドしたい場合があります。
  • デフォルトの引数値は、まれな例外のために多くの関数を定義する必要なく、これを行う簡単な方法を提供します。 Python はオーバーロードされたメソッド/関数をサポートしていないため、デフォルトの引数はオーバーロード動作を「偽装」する簡単な方法です。

デメリット

  • デフォルトの引数は、モジュールのロード時に 1 回評価されます。 引数がリストや辞書などの変更可能なオブジェクトである場合、これは問題を引き起こす可能性があります。 関数がオブジェクトを変更する場合 (たとえば、項目をリストに追加することによって)、デフォルト値が変更されます。

結論

  • 次の注意事項を守って使用してください。
  • 関数またはメソッドの定義で、可変オブジェクトをデフォルト値として使用しないでください。

Properties

Tips

  • プロパティは、簡単な計算やロジックを必要とする属性の取得または設定を制御するために使用できます。 プロパティの実装は、通常の属性アクセスの一般的な期待に一致する必要があります。つまり、安価で、簡単で、驚くべきことではありません。

定義

  • 属性を取得および設定するためのメソッド呼び出しを標準の属性アクセスとしてラップする方法。

メリット

  • getter および setter メソッドの呼び出しではなく、属性アクセスおよび割り当て API を許可します。
  • 属性を読み取り専用にするために使用できます。
  • 計算を遅延させることができます。
  • 内部がクラス ユーザーとは独立して進化するときに、クラスのパブリック インターフェイスを維持する方法を提供します。

デメリット

  • 演算子のオーバーロードと同じように、副作用を隠すことができます。
  • サブクラスを混乱させる可能性があります。

結論

  • プロパティは許可されますが、演算子のオーバーロードと同様に、必要な場合にのみ使用し、通常の属性アクセスの期待に一致させる必要があります。 それ以外の場合は、ゲッターとセッターの規則に従います。
  • たとえば、内部属性を単に取得および設定するためにプロパティを使用することは許可されていません。計算は行われないため、プロパティは不要です (代わりに属性をパブリックにします)。 それに比べて、プロパティを使用して属性アクセスを制御したり、単純に導出された値を計算したりすることは許可されています。ロジックは単純で驚くべきことではありません。
  • プロパティは @property デコレータで作成する必要があります。 プロパティ記述子を手動で実装することは、強力な機能と見なされます。
  • プロパティによる継承は、明白ではない場合があります。 サブクラスがオーバーライドおよび拡張する可能性のある計算を実装するためにプロパティを使用しないでください。
