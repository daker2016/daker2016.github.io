Google Python Style Guide - Python Style Rules

Python Style Rules

Semicolons

Tips

  • 行末にコンマを使わないでください。二つステートメント間にもコンマを使わないでください。

Line length

Tips

  • 最大80文字

例外

  • 長いimportステートメント
  • URL、パス名、コメント中に長いflags
  • URL やパス名など、複数の行に分割するのが不便な空白を含まない長い文字列モジュール レベルの定数。
    • Pylintはコメントを無効にする (e.g.: # pylint: disable=invalid-name)
  • withステートメント、3つ以上のコンテキストマネージャーを必要とするを除き、バックスラッシュの行継続を使用しないでください。
  • 括弧、中かっこ、大かっこを使う。 必要に応じて、式の周りに括弧のペアを追加できます。
  • リテラル文字列が 1 行に収まらない場合は、括弧を使用して暗黙的な行結合を行います。
  • コメント内では、必要に応じて長い URL を独自の行に入れます。
  • 式が 3 行以上にまたがる with ステートメントを定義するときは、バックスラッシュの継続を使用できます。 2 行の式の場合は、ネストされた with ステートメントを使用します。
  • 上記の行継続の例の要素のインデントに注意してください。 説明については、インデントのセクションを参照してください。
  • 行が 80 文字を超え、yapf オートフォーマッターが行を制限以下にするのに役立たないその他のすべてのケースでは、行はこの最大値を超えることができます。 賢明な場合、著者は上記の注記に従って手動でラインアップを分割することをお勧めします。

Parentheses（丸括弧）

Tips

  • 括弧は慎重に使用してください
  • 必須ではありませんが、tuplesを括弧で囲んでも問題ありません。 暗黙の行継続またはtupleを示すために括弧を使用しない限り、return ステートメントまたは条件ステートメントでそれらを使用しないでください。

Indentation

Tips

  • 4つのスペースを使って下さい
  • Tabを使わないでください。行継続の場合、ラップされた要素を垂直方向に揃える必要があります

Trailing commas in sequences of items

Tips

  • 一連の項目の末尾のコンマは、終了コンテナートークン]、)、または } が最後の要素と同じ行に表示されない場合にのみ推奨されます。末尾のコンマの存在は、Python コードの自動フォーマッタ YAPF へのヒントとしても使用され、最後の要素の後に , が存在する場合に、アイテムのコンテナを 1 行に 1 つのアイテムに自動フォーマットするように指示します。

Blank Lines

Tips

  • 関数定義かクラス定義かに関係なく、最上位の定義の間には 2 つの空白行があります。 メソッド定義の間、およびクラス行と最初のメソッドの間に 1 行の空白行。
  • def 行の後に空白行はありません。 関数またはメソッド内で適切と判断した場合は、単一の空白行を使用してください。
  • 空行を定義に固定する必要はありません。たとえば、関数、クラス、およびメソッドの定義の直前にある関連するコメントは意味があります。 あなたのコメントが docstring の一部として役立つかどうかを検討してください。

Whitespace

Tips

  • 句読点の前後のスペースの使用については、標準の印刷規則に従ってください。
  • ()、[]、{}内部に空白がありません
  • コンマ、セミコロン、またはコロンの前に空白はありません。 行末を除いて、コンマ、セミコロン、またはコロンの後に空白を使用してください。
  • 引数リスト、インデックス付け、またはスライスを開始する開き括弧/ブラケットの前に空白はありません。
  • 末尾の空白はありません。
  • 二項演算子は、代入 (=)、比較 (==、<、>、!=、<>、<=、>=、in、not in、is、is not)、およびブール値の両側を 1 つのスペースで囲みます。 （そして、または、そうではありません）。 算術演算子 (+、-、*、/、//、%、**、@) の前後にスペースを挿入する場合は、適切に判断してください。
  • キーワード引数を渡すとき、または既定のパラメーター値を定義するときに = の周りにスペースを使用しないでください。1 つの例外があります。型注釈が存在する場合は、既定のパラメーター値の=の周りにスペースを使用してください。
  • メンテナンスの負担になるため、連続する行でトークンを垂直方向に揃えるためにスペースを使用しないでください (:、#、= などに適用されます)。

Shebang Line

Tips

  • ほとんどの .py ファイルは #! で始める必要はありません。 ライン。 プログラムのメイン ファイルを #!/usr/bin/env python3 (virtualenvs をサポートするため) または #!/usr/bin/python3 per PEP-394 で開始します。
  • この行は、Python インタープリターを見つけるためにカーネルによって使用されますが、モジュールをインポートするときに Python によって無視されます。 直接実行することを意図したファイルでのみ必要です。

Comments and Docstrings

Tips

  • モジュール、関数、メソッドのドキュメント文字列、およびインライン コメントには、必ず適切なスタイルを使用してください。

Strings

Tips

  • パラメーターがすべて文字列の場合でも、文字列の書式設定には f-string、% 演算子、または format メソッドを使用します。 + と文字列のどちらの形式にするかは、最善の判断で決定してください。

Files, Sockets, and similar Stateful Resources

Tips

  • ファイルとソケットの使用が終了したら、それらを明示的に閉じます。このルールは、データベース接続など、ソケットを内部で使用するクローズ可能なリソースや、同様の方法でクローズする必要があるその他のリソースにも自然に拡張されます。ほんの数例を挙げると、これには mmapマッピング、 h5py ファイル オブジェクト、および matplotlib.pyplot Figure ウィンドウも含まれます。
  • ファイル、ソケット、またはその他のステートフル オブジェクトを不必要に開いたままにしておくと、多くの欠点があります。
    • ファイル記述子など、限られたシステム リソースを消費する可能性があります。このようなオブジェクトを多数処理するコードは、使用後にすぐにシステムに返されない場合、これらのリソースを不必要に使い果たす可能性があります。
    • ファイルを開いたままにしておくと、ファイルの移動や削除、ファイルシステムのアンマウントなどの他のアクションが妨げられる場合があります。
    • プログラム全体で共有されているファイルとソケットは、論理的に閉じられた後に、誤って読み書きされてしまう可能性があります。それらが実際に閉じられている場合、それらから読み書きしようとすると例外が発生し、問題がすぐにわかります。
  • さらに、ファイルとソケット (および同様に動作するリソース) は、オブジェクトが破棄されると自動的に閉じられますが、オブジェクトの有効期間をリソースの状態に結び付けるのは適切ではありません。
    • ランタイムが実際にいつ __del__メソッドを呼び出すかについての保証はありません。Python の実装が異なれば、遅延ガベージ コレクションなどのさまざまなメモリ管理手法が使用されます。これにより、オブジェクトの有効期間が任意かつ無期限に長くなる可能性があります。
    • ファイルへの予期しない参照 (グローバルや例外のトレースバックなど) により、意図したよりも長くファイルが保持される場合があります。
  • ファイナライザーに依存して自動クリーンアップを実行すると、目に見える副作用が発生することが何度も再発見され、何十年にもわたって複数の言語で大きな問題が発生しています ( Javaについては、この記事を参照してください)。

TODOコメント

Tips

  • TODO一時的なコード、短期的な解決策、または十分ではあるが完全ではないコードにはコメントを使用します。
  • TODOコメントは、すべて大文字の単語とTODO、括弧で囲まれたコンテキスト識別子で始まります。理想的にはバグ リファレンス、場合によってはユーザー名。バグは追跡され、フォローアップ コメントがあるため、バグ リファレンスTODO(https://crbug.com/bug_id_number):は望ましいものです。続いて、何をすべきかのTODO説明が続きます。
  • TODO目的は、より詳細な情報を取得する方法を見つけるために検索できる一貫した形式にすることです。ATODOは、参照された人物が問題を解決するというコミットメントではありません。したがってTODO、ユーザー名を使用して を作成する場合、ほとんどの場合、指定されるのは自分のユーザー名です。
  • TODO「将来の日付で何かを行う」という形式の場合は、非常に具体的な日付 (「2009 年 11 月までに修正」) または非常に具体的なイベント (「すべてのクライアントが XML 応答を処理できるようになったら、このコードを削除してください。」) 将来のコード管理者が理解するでしょう。

ステートメント

Tips

  • 通常、1 行に 1 つのステートメントのみです。
  • ただし、ステートメント全体が 1 行に収まる場合に限り、テストの結果をテストと同じ行に入れることができます。特に、 と の両方を同じ行に収めることはできないため、try/を使用してこれを行うことはできません。except try except if else

ゲッターとセッター

Tips

  • ゲッター関数とセッター関数 (アクセサーおよびミューテーターとも呼ばれる) は、変数の値を取得または設定するための意味のある役割または動作を提供する場合に使用する必要があります。
  • 特に、変数の取得または設定が複雑な場合、または現在または合理的な将来においてコストが大きい場合に使用する必要があります。
  • たとえば、getter/setter のペアが単純に内部属性を読み書きする場合は、代わりに内部属性を公開する必要があります。比較すると、変数の設定が何らかの状態の無効化または再構築を意味する場合、それはセッター関数である必要があります。関数呼び出しは、潜在的に重要な操作が発生していることを示唆しています。または、単純なロジックが必要な場合や、getter と setter を不要にするリファクタリングが必要な場合は、プロパティがオプションになる場合があります。
  • ゲッターとセッターは、 や などの命名get_foo()ガイドラインに従う必要がありset_foo()ます。
  • 以前の動作でプロパティを介したアクセスが許可されていた場合は、新しい getter/setter 関数をプロパティにバインドしないでください。古い方法で変数にアクセスしようとしているコードは、複雑さの変化を認識できるように、目に見えて中断する必要があります。

ネーミング

Tips

  • module_name, package_name, ClassName, method_name, ExceptionName, function_name, GLOBAL_CONSTANT_NAME, global_var_name, instance_var_name, . function_parameter_name_ local_var_name_ query_proper_noun_for_thing_ send_acronym_via_https
  • 関数名、変数名、およびファイル名は、わかりやすいものにする必要があります。略語は避けてください。特に、プロジェクト外の読者にとってあいまいまたはなじみのない略語を使用したり、単語内の文字を削除して略語を使用したりしないでください。
  • 常に.pyファイル拡張子を使用してください。ダッシュは絶対に使用しないでください。

避けるべき名前

  • 特別に許可されている場合を除き、1 文字の名前:
    • カウンタまたはイテレータ (例: i、j、k、vなど)
    • try/exceptステートメントの例外識別子として。
    • withステートメント内のファイル ハンドルとして
    • 制約TypeVarのないプライベート(例: _T、 _U)_V
      1 文字のネーミングを乱用しないように注意してください。一般的に言えば、説明性は名前の可視性の範囲に比例する必要があります。たとえば、i5 行のコード ブロックには適切な名前かもしれませんが、複数のネストされたスコープ内では、あいまいすぎる可能性があります。
  • ダッシュ(-)in パッケージ名またはモジュール名
  • __double_leading_and_trailing_underscore__名
  • 不快な言葉
  • 変数の型を不必要に含む名前 (例: id_to_name_dict)

命名規則

  • 「内部」とは、モジュールの内部、またはクラス内の保護またはプライベートを意味します。
  • 単一のアンダースコア ( _) を先頭に追加すると、モジュール変数と関数を保護するためのサポートが得られます (リンターは、保護されたメンバー アクセスにフラグを立てます)。
  • インスタンス変数またはメソッドの先頭に 2 つのアンダースコア (__別名「dunder」) を追加すると、変数またはメソッドがそのクラスに対して効果的にプライベートになります (名前マングリングを使用)。可読性とテスト容易性に影響を与え、実際には非公開ではないため、使用しないことをお勧めします。単一のアンダースコアを優先します。
  • 関連するクラスとトップレベルの関数をまとめてモジュールに配置します。Java とは異なり、モジュールごとに 1 つのクラスに制限する必要はありません。
  • クラス名には CapWords を使用しますが、モジュール名には lower_with_under.py を使用します。CapWords.py という名前の古いモジュールがいくつかありますが、モジュールがたまたまクラスにちなんで名付けられていると混乱するため、これは推奨されません。（「待って–私が書いたのですimport StringIOかfrom StringIO import StringIO？」）
  • アンダースコアは、名前の論理コンポーネントを区切るために、 で始まるunittestメソッド名に表示される場合がありtestます。それらのコンポーネントが CapWords を使用している場合でも同様です。考えられるパターンの 1 つは次のとおりですtest_。たとえばtestPop_EmptyStack大丈夫です。テスト メソッドに名前を付ける正しい方法は 1 つではありません。

数学表記

関数の長さ

Tips

  • 小さくて焦点を絞った機能を好む。
  • 長い関数が適切な場合があることを認識しているため、関数の長さに厳密な制限はありません。関数が約 40 行を超える場合は、プログラムの構造を損なわずに分割できるかどうかを検討してください。
  • 長い関数が今は完全に機能していても、数か月後に誰かがそれを変更すると、新しい動作が追加される可能性があります。これにより、見つけにくいバグが発生する可能性があります。関数を短くシンプルにしておくと、他の人がコードを読んだり変更したりしやすくなります。
  • コードを操作していると、長くて複雑な関数を見つけることがあります。既存のコードを変更することをためらわないでください。そのような関数を使用するのが難しいことが判明した場合、エラーをデバッグするのが難しい場合、またはいくつかの異なるコンテキストでその一部を使用したい場合は、関数をより小さなコードに分割することを検討してください。そしてより扱いやすい部分。

一般規則

  • PEP-484に慣れてください 。

  • 他の変数または戻り型を表現しない場合は、 を使用しますAny。
  • モジュール内のすべての関数に注釈を付ける必要はありません。
    • 少なくとも公開 API に注釈を付けてください。
    • 判断力を働かせて、一方では安全性と明快さ、他方では柔軟性との間でバランスをとってください。
    • 型関連のエラー (以前のバグまたは複雑さ) が発生しやすいコードに注釈を付けます。
    • 理解しにくいコードに注釈を付けます。
    • 型の観点からコードが安定するようになると、コードに注釈を付けます。多くの場合、柔軟性をあまり失うことなく、成熟したコードのすべての関数に注釈を付けることができます。

改行

  • 既存のインデント規則に従うようにしてください。
  • 注釈を付けた後、多くの関数シグネチャは「1 行に 1 つのパラメーター」になります。戻り値の型にも独自の行が指定されるようにするには、最後のパラメーターの後にコンマを配置できます。
  • たとえば、変数名と型注釈の間ではなく、常に変数の間で分割することを優先します。ただし、すべてが同じラインに収まる場合は、それを選択してください。
  • 関数名、最後のパラメーター、および戻り値の型の組み合わせが長すぎる場合は、新しい行で 4 だけインデントします。改行を使用する場合は、各パラメーターと戻り値の型をそれぞれの行に配置し、閉じ括弧を次のように揃えることをお勧めしますdef。
  • 必要に応じて、戻り値の型を最後のパラメーターと同じ行に置くことができます。
  • pylint閉じ括弧を新しい行に移動して、開き括弧と揃えることができますが、これは読みにくくなります。
  • 上記の例のように、型を壊さないことを好みます。ただし、長すぎて 1 行に収まらない場合があります (サブタイプが途切れないようにしてください)。
  • 単一の名前とタイプが長すぎる場合は 、タイプのエイリアスを使用することを検討してください。最後の手段は、コロンの後で区切り、4 ずつインデントすることです。
